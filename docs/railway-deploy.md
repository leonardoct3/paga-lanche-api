# Railway Deploy

Railway can deploy this API directly from GitHub. No GitHub Actions workflow is
needed for the basic setup: Railway watches the selected branch and deploys new
commits automatically.

## 1. Prepare The Repository

Make sure these files are committed:

- `Dockerfile`
- `requirements.txt`
- `src/`
- `.env.example`

Do not commit `.env`.

## 2. Create The Railway Project

1. Go to Railway.
2. Create a new project.
3. Choose `Deploy from GitHub repo`.
4. Select this repository.
5. Railway should detect and use the root `Dockerfile`.

The Dockerfile uses:

```bash
uvicorn src.main:app --host 0.0.0.0 --port ${PORT:-8000}
```

Railway provides `PORT` automatically, so no special start command is needed.

## 3. Add PostgreSQL

Inside the same Railway project:

1. Click `+ New`.
2. Choose `Database`.
3. Choose `PostgreSQL`.

Railway provides database variables for the Postgres service, including:

- `DATABASE_URL`
- `PGHOST`
- `PGPORT`
- `PGUSER`
- `PGPASSWORD`
- `PGDATABASE`

This API only needs `DATABASE_URL`.

## 4. Configure API Variables

Open the API service, then add variables:

```env
API_KEY=use-a-strong-production-key
DATABASE_URL=${{Postgres.DATABASE_URL}}
```

Do not use the local Docker Compose value in Railway:

```env
DATABASE_URL=postgresql+psycopg://paga_lanche:paga_lanche@db:5432/paga_lanche
```

The host `db` only exists when running `docker compose` locally. Railway needs
the URL generated by the Railway PostgreSQL service.

The exact `Postgres` service name in the template reference depends on what the
database service is called in your Railway project. If Railway named it
`Postgres`, the example above works. If you renamed it, use that service name.

You can also copy the generated `DATABASE_URL` from the Postgres service and
paste it directly into the API service variables.

## 5. Generate A Public Domain

Open the API service:

1. Go to `Settings`.
2. Go to `Networking`.
3. Generate a public domain.

After deployment, test:

```text
https://your-domain.up.railway.app/health
```

Swagger will be available at:

```text
https://your-domain.up.railway.app/docs
```

## 6. Use The API Key In Swagger

1. Open `/docs`.
2. Click `Authorize`.
3. Enter your `API_KEY`.
4. Try a protected endpoint like `GET /users`.

## 7. Automatic Deploys

Railway services linked to a GitHub repo automatically deploy when new commits
are pushed to the connected branch.

You can change the deploy branch in the service settings.

For this project, the recommended branch is:

```text
main
```

## 8. Common Problems

If the build fails, check that Railway is using the `Dockerfile` at the repo
root.

If the app deploys but crashes, check:

- `DATABASE_URL` exists in the API service variables.
- `DATABASE_URL` does not contain `@db:5432`.
- `API_KEY` exists in the API service variables.
- The Postgres service is running.
- The API logs do not show a database connection error.

If logs show this:

```text
failed to resolve host 'db'
```

Replace the API service `DATABASE_URL` with Railway's Postgres variable:

```env
DATABASE_URL=${{Postgres.DATABASE_URL}}
```

If protected endpoints return `401`, the `X-API-Key` header is missing or does
not match the `API_KEY` variable.

## 9. Cost Control

For the $5 Railway plan, keep the setup small:

- one API service;
- one PostgreSQL service;
- no extra preview environments at first;
- avoid external database connections unless you need them.

Watch usage during the first few days after launch.
